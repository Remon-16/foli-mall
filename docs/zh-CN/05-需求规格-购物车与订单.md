## 二、业务范围与角色定义

### 2.1 名词解释

| 术语 | 说明 |
|------|------|
| 购物车去重 | 同一用户添加相同商品时，数量累加而非创建新行 |
| 选中结算 | 购物车中标记为 selected=1 的商品参与下单 |

### 2.3 需求范围

**本次做**：购物车 CRUD（去重累加）；下单并余额支付（乐观锁扣库存+扣余额）；订单生命周期（支付→发货→收货→完成→取消）；余额流水记录

**本次不做**：
- 先下单后付款（PENDING_PAY 定义但非当前主线——创建即支付）
- 跨店铺购物车拆分（不检查购物车商品是否同属一家店）
- RECEIVED(3) 中间状态（当前直接从 SHIPPED 跳到 COMPLETED）

## 三、业务流程图

### 3.1 订单状态机

```
  createOrder
       │
       ▼
  ┌──────────┐     payOrder       ┌──────────┐
  │PENDING_PAY│ ─────────────────→ │   PAID   │
  │    (0)    │                    │   (1)    │
  └─────┬─────┘                    └────┬─────┘
        │ cancelOrder                   │ shipOrder
        ▼                               ▼
  ┌──────────┐                    ┌──────────┐
  │CANCELLED │                    │ SHIPPED  │
  │   (5)    │                    │   (2)    │
  └──────────┘                    └────┬─────┘
                                       │ receiveOrder
                                       ▼
                                  ┌──────────┐
                                  │COMPLETED │
                                  │   (4)    │
                                  └──────────┘

  RECEIVED(3) — 已定义，但当前流程中未使用（SHIPPED 直接跳到 COMPLETED）
  PENDING_PAY(0) — 已定义，但 createOrder 创建时即为 PAID（一体化下单支付）
```

**关键规则**：
- `createOrder` 创建时即为 PAID 状态（跳过 PENDING_PAY），已扣余额
- `payOrder` 和 `cancelOrder` 仅对 PENDING_PAY 订单有效（当前流程中不可达）
- 各状态转换均有严格的状态机校验

### 3.2 下单支付时序

```
买家 → 获取选中购物车项
     → 遍历验证：商品存在 + APPROVED + 库存充足(>=qty)
     → 乐观锁扣库存：UPDATE stock = stock - qty WHERE stock >= qty
     → 乐观锁扣余额：UPDATE balance = balance - total WHERE balance >= total
     → 创建订单（PAID 状态，记录 payTime）
     → 创建订单项（快照：商品名、图片、价格）
     → 更新商品销量
     → 记录余额流水（PAY 类型）
     → 清空已选购物车项
```

## 四、功能需求详细说明

### 4.1 购物车管理

**添加商品**：
- 前置条件：商品存在 且 status=APPROVED 且 stock>0
- 去重：同 productId 已存在时累加 quantity，不新增行
- 默认 quantity=1，selected=1（默认参与结算）

**更新购物车**：
- 可修改数量（quantity）和选中状态（selected）
- 更新时不验证库存（可能出现过期数据）

**删除**：单个删除或清空均为逻辑删除

**异常分支**：商品不存在(203001) / 商品已下架(203004) / 库存不足(203002) / 购物车项不存在(207002)

### 4.2 下单

**前置条件**：已登录，购物车中有选中商品。

**业务规则**：
- 获取所有 selected=1 的购物车项
- 验证每个商品：存在 + APPROVED + 库存>=购买数量
- 原子扣库存：`UPDATE SET stock = stock - qty WHERE stock >= qty AND id = ?`
- 原子扣余额：`UPDATE SET balance = balance - total WHERE balance >= total AND id = ?`
- 订单初始状态为 PAID(1)（一体化支付）
- 记录余额流水（PAY 类型，含前后余额）
- 更新商品销量
- 清空已选中购物车项

**异常分支**：购物车空(207001) / 商品不存在(203001) / 商品下架(203004) / 库存不足(203002) / 扣库失败(203003) / 余额不足(205001) / 扣款失败(205002)

### 4.3 支付、取消、发货、收货

| 操作 | 前置状态 | 目标状态 | 操作者 | 附加动作 |
|------|---------|---------|--------|---------|
| 支付 | PENDING_PAY | PAID | 买家（自己的订单） | 扣余额+日志 |
| 取消 | PENDING_PAY | CANCELLED | 买家（自己的订单） | 退余额+恢复库存+日志 |
| 发货 | PAID | SHIPPED | 卖家（自己的店铺） | 记录 shipTime |
| 收货 | SHIPPED | COMPLETED | 买家（自己的订单） | 记录 receiveTime+completeTime |

### 4.4 订单号规则

格式：`FO` + `yyyyMMdd`（8 位日期） + 6 位随机数字

示例：`FO20260624123456`

## 五、非功能需求

### 5.1 并发安全

- 库存扣减：乐观锁（WHERE stock >= qty），防止超卖
- 余额扣减：乐观锁（WHERE balance >= amount），防止超扣
- 事务回滚：两者任一失败则整体回滚（@Transactional）

### 5.2 数据一致性

- 下单在单个事务中完成：验证→扣库存→扣余额→建订单→清购物车→记流水
- 取消订单也在事务中：恢复库存+退还余额+记录退款流水
