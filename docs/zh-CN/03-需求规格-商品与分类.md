## 三、业务流程图

### 3.1 商品状态机

```
  publishProduct / updateProduct / onShelf
         │
         ▼
  ┌──────────────────┐
  │ PENDING_REVIEW(1)│ ← 编辑商品和重新上架也会进入此状态
  │     待审核        │
  └────┬─────────┬───┘
       │         │
  APPROVED   REJECTED
       │         │
       ▼         ▼
  ┌─────────┐  ┌──────────┐
  │APPROVED │  │REJECTED  │
  │  (2)    │  │   (3)    │
  └───┬─────┘  └──────────┘
      │ offShelf
      ▼
  ┌──────────┐
  │OFF_SHELF │── onShelf ──→ PENDING_REVIEW (重新审核)
  │   (4)    │
  └──────────┘
```

**关键规则**：
- 发布商品直接进入 PENDING_REVIEW（无草稿流程）
- 编辑商品强制重置为 PENDING_REVIEW
- 仅 PENDING_REVIEW 状态可被审核（状态机校验已实现）
- 下架无前置状态限制（任何状态均可下架）
- 重新上架进入 PENDING_REVIEW（不会直接恢复 APPROVED）

### 3.2 商品搜索流程

仅返回 APPROVED 商品 → 关键词模糊匹配商品名 → 分类筛选含所有子分类 → 价格区间 → 排序(newest/price_asc/price_desc/sales) → 分页返回

## 四、功能需求详细说明

### 4.1 分类管理

**使用场景**：管理员维护商品分类体系。

**业务规则**：树形结构（parentId=0 为顶级）；当前版本两级；仅 status=1 的分类出现在分类树；有子分类的父分类不可删除

**异常分支**：分类不存在(211001) / 父分类设为自身(211003) / 有子分类时删除(211002)

### 4.2 商品发布

**前置条件**：SELLER 角色，拥有已审核通过的店铺。

**业务规则**：名称非空；价格>0；分类必选；库存默认0；初始销量0；初始状态 PENDING_REVIEW；多张详情图按 sortOrder 排序

### 4.3 商品编辑

**前置条件**：是商品所属店铺的卖家。

**业务规则**：验证所有权（storeId 匹配）；编辑后状态重置为 PENDING_REVIEW；清除旧审核意见；旧图删除、新图重新保存

**异常分支**：商品不存在(203001) / 无权操作(200003)

### 4.4 商品审核

**前置条件**：ADMIN 角色。

**业务规则**：仅 PENDING_REVIEW 可审核（状态机校验）；审核状态值只接受 APPROVED(2) 或 REJECTED(3)

**异常分支**：商品不存在(203001) / 已审核完毕(203005) / 无效审核状态值(202004)

### 4.5 商品搜索与详情

**搜索**：仅返回 APPROVED 商品；关键词 LIKE 模糊匹配；分类筛选含递归子分类；支持价格区间和排序

**详情**：不限制状态（任何人可查看任何状态商品的详情）；填充店铺名和分类名

## 五、非功能需求

- 卖家只能操作自己店铺的商品（storeId 所有权验证）
- 搜索时子分类递归在内存中完成（一次性加载所有分类）
- 店铺名称批量填充（一次 IN 查询，避免 N+1 问题）
