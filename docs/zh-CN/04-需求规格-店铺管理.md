## 三、业务流程图

### 3.1 店铺状态机

```
  applyStore / updateStore
         │
         ▼
  ┌──────────────┐
  │ PENDING (0)  │ ← 更新店铺信息也会重置为此状态
  │    待审核     │
  └──┬──────┬────┘
     │      │
  APPROVED REJECTED
     │      │
     ▼      ▼
  ┌────────────┐  ┌──────────────┐
  │APPROVED (1)│  │ REJECTED (2) │
  │  已通过     │  │   已拒绝      │
  └────────────┘  └──────────────┘
```

**CLOSED(3)** 状态已定义但从未使用——无关闭店铺功能。

**关键规则**：
- 申请开店后状态为 PENDING（待审核）
- 更新店铺信息后强制重置为 PENDING（需重新审核）
- 审核时检查当前状态为 PENDING（已实现状态机校验）
- 审核只接受 APPROVED(1) 或 REJECTED(2)

### 3.2 开店申请流程

```
用户申请开店
  → 验证用户存在
  → 若角色为 BUYER → 自动升级为 SELLER
  → 若角色为 ADMIN → 拒绝（管理员不能开店）
  → 检查该用户是否已有活跃店铺（userId + isDelete=0）
  → 已有店铺 → 拒绝申请（每用户限 1 个店铺）
  → 创建店铺记录，初始状态 PENDING

管理员审核店铺
  → 检查店铺状态为 PENDING
  → 选择通过(APPROVED) 或拒绝(REJECTED)
  → 可附审核意见
```

## 四、功能需求详细说明

### 4.1 开店申请

**使用场景**：用户申请开设自己的店铺（BUYER 用户申请时自动升级为 SELLER）。

**前置条件**：已登录。

**业务规则**：
- BUYER 用户申请开店时自动升级为 SELLER 角色
- ADMIN 用户不能申请开店
- 每个用户最多拥有 1 个活跃店铺（isDelete=0）
- 店铺初始状态为 PENDING(0)
- 店铺名称非空

**异常分支**：
- 用户不存在 → 201004
- 非 BUYER/SELLER 角色（ADMIN） → 202005
- 已有店铺 → 202001

### 4.2 更新店铺信息

**前置条件**：是店铺的所有者（userId 匹配）。

**业务规则**：
- 任何状态均可更新（包括已拒绝的）
- 更新后状态重置为 PENDING，需重新审核
- 清除旧的审核意见

**异常分支**：店铺不存在(202003) / 无权操作(200003)

### 4.3 店铺审核

**前置条件**：ADMIN 角色。

**业务规则**：
- 仅 PENDING 状态的店铺可审核（状态机校验）
- 审核状态值只接受 APPROVED(1) 或 REJECTED(2)
- 可附审核意见

**异常分支**：店铺不存在(202003) / 已审核完毕(202002) / 无效审核状态(202004)

### 4.4 公开浏览

**业务规则**：公开列表仅返回 APPROVED 状态店铺；店铺详情不限制状态（含审核意见）；返回店主昵称和商品数量
