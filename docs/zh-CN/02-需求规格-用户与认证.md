## 二、业务范围与角色定义

### 2.1 名词解释

| 术语 | 说明 |
|------|------|
| JWT Token | 用户登录后签发的无状态令牌，包含 userId、role、username，有效期 24 小时 |
| BCrypt | 密码哈希算法，用于注册时加密和登录时校验 |
| ThreadLocal 上下文 | 拦截器从 Token 中提取用户信息后存入 ThreadLocal，请求结束自动清理 |
| 余额 | 用户账户中的可用资金，用于下单支付 |

### 2.2 角色与权限

本模块中三种角色的权限：

| 功能 | BUYER | SELLER | ADMIN |
|------|-------|--------|-------|
| 注册 | 可（默认注册为 BUYER） | — | — |
| 登录 | 可 | 可 | 可 |
| 查看个人信息 | 可（自己的） | 可（自己的） | 可（自己的） |
| 查询余额 | 可（自己的） | 可（自己的） | 可（自己的） |
| 充值 | 可（自己的） | 可（自己的） | 可（自己的） |
| 查看余额流水 | 可（自己的） | 可（自己的） | 可（自己的） |
| 禁用/启用用户 | — | — | 可 |

### 2.3 需求范围

**本次做**：
- 用户名+密码注册，默认 BUYER 角色
- BCrypt 加密存储密码
- JWT 登录认证，返回 Token
- 基于注解的角色权限控制（@RequireLogin, @RequireRole）
- 账户余额查询与充值
- 余额变动流水记录

**本次不做**：
- 手机号/邮箱验证
- 第三方登录（OAuth）
- 密码找回/重置
- 多因素认证（MFA）
- Token 刷新机制

## 三、业务流程图

### 3.1 用户注册流程

```
用户提交注册信息
  → 参数校验：用户名 3-50 字符，密码 6-100 字符
  → 检查用户名唯一性（数据库 UNIQUE 约束）
  → BCrypt 加密密码
  → 创建用户记录（默认角色=BUYER，余额=0，状态=启用）
  → 返回用户信息（不含密码）
```

### 3.2 用户登录流程

```
用户提交登录信息
  → 查询用户名
  → 用户名不存在 → 返回"用户名或密码错误"（不区分具体原因）
  → BCrypt 验证密码
  → 密码不匹配 → 返回"用户名或密码错误"
  → 检查账号状态：status=0 → 返回"账号已被禁用"
  → 签发 JWT Token（负载：userId, role, username，有效期 24h）
  → 返回 Token 和用户信息
```

### 3.3 鉴权流程

```
请求到达
  → 检查是否需要登录（类/方法上是否有 @RequireLogin）
  → 公开接口 → 直接放行
  → 需要登录 → 检查 Authorization header（Bearer Token）
  → 无 Token 或格式错误 → HTTP 401
  → 校验 Token（签名 + 过期）
  → Token 无效或过期 → HTTP 401
  → 提取 userId, role, username 存入 UserContext (ThreadLocal)
  → 检查角色限制（@RequireRole）
  → 角色不匹配 → HTTP 403
  → 放行
  → 请求结束后清理 ThreadLocal
```

### 3.4 充值流程

```
用户提交充值金额
  → 参数校验：金额 > 0
  → 查询当前余额（变动前余额）
  → 原子更新余额：UPDATE fm_user SET balance = balance + 金额 WHERE id = userId
  → 记录余额流水（type=RECHARGE，含变动前后余额）
```

## 四、功能需求详细说明

### 4.1 用户注册

**使用场景**：新用户首次使用系统时创建账号。

**前置条件**：无。

**业务规则**：
- 用户名长度 3-50 字符，全局唯一（数据库 UNIQUE 约束）
- 密码长度 6-100 字符
- 密码使用 BCrypt 算法单向哈希存储
- 新用户默认角色为 BUYER（编码 0）
- 新用户默认为启用状态（status=1）
- 初始余额为 0.00
- 昵称可选，不填则默认使用用户名

**异常分支**：
- 用户名已存在 → 业务码 201001 "用户名已存在"
- 参数校验失败 → HTTP 400

### 4.2 用户登录

**使用场景**：已注册用户获取访问凭证。

**前置条件**：已注册且账号未被禁用。

**业务规则**：
- 用户名不存在或密码错误均返回相同提示，防止用户名枚举
- 账号被禁用（status=0）时拒绝登录
- JWT Token 有效期 24 小时，使用 HS256 签名
- Token 负载包含：userId, role, username

**异常分支**：
- 用户名或密码错误 → 业务码 201002 "用户名或密码错误"
- 账号已被禁用 → 业务码 201003 "账号已被禁用"

### 4.3 账户余额查询

**使用场景**：用户查看自己的账户余额。

**前置条件**：已登录。

**业务规则**：
- 仅可查询自己的余额
- 余额为 null 时返回 0.00

**异常分支**：
- 用户不存在 → 业务码 201004 "用户不存在"

### 4.4 账户充值

**使用场景**：用户为自己的账户充值。

**前置条件**：已登录。

**业务规则**：
- 充值金额必须大于 0
- 使用原子 UPDATE 语句增加余额
- 记录余额流水（type=RECHARGE，包含变动前/后余额）
- 无充值上限限制

**异常分支**：
- 用户不存在 → 业务码 201004 "用户不存在"
- 余额更新失败（并发冲突）→ 业务码 205003 "充值失败，余额更新异常"

### 4.5 余额流水查询

**使用场景**：用户查看自己的余额变动记录。

**前置条件**：已登录。

**业务规则**：
- 仅可查看自己的流水
- 按创建时间倒序排列
- 支持分页

### 4.6 用户状态管理（管理员）

**使用场景**：管理员启用或禁用用户账号。

**前置条件**：已登录，角色为 ADMIN。

**业务规则**：
- 设置 status=0（禁用）或 status=1（启用）
- 禁用的账号无法登录

**异常分支**：
- 用户不存在 → 业务码 201004 "用户不存在"

## 五、非功能需求

### 5.1 安全需求

- 密码绝不以明文存储或传输
- JWT 密钥可配置（默认值仅供开发使用）
- 登录失败时不区分"用户名不存在"和"密码错误"，防止用户枚举
- ThreadLocal 在请求结束后强制清理，防止内存泄漏

### 5.2 性能需求

- 密码 BCrypt 哈希在注册和登录时执行，属于 CPU 密集型操作，但单次耗时 <100ms
- 余额充值为单表单行 UPDATE，毫秒级完成
