### 1.2 项目背景

在 API 自动化测试领域，测试用例的生成和验证需要依赖一个功能完备、状态可预测的被测系统作为测试靶场。市面上的开源电商系统往往过于复杂，部署成本高，且状态不易重置。因此需要构建一个轻量级、零外部依赖、快速启动的商城系统，专门用作 API 测试用例生成智能体的测试靶场。

### 1.3 项目目标

- **零外部依赖**：使用 H2 内存数据库，无需安装任何外部中间件
- **快速启动**：前后端均可在数秒内启动，支持热重载
- **功能完备**：覆盖电商核心业务流程——用户注册/登录、商品搜索、购物车、下单支付、退货退款、投诉处理
- **状态可重置**：重启后端即恢复初始数据状态，便于反复测试
- **角色分离**：内置买家、卖家、管理员三种角色，覆盖电商三方协作场景

### 1.4 术语与缩写

| 术语 | 说明 |
|------|------|
| Foli Mall | 福礼商城，本项目名称 |
| JWT | JSON Web Token，无状态认证令牌 |
| BCrypt | 跨平台密码哈希算法 |
| H2 | 轻量级 Java 内存数据库，MySQL 兼容模式 |
| RBAC | 基于角色的访问控制 (Role-Based Access Control) |
| 雪花 ID | Twitter Snowflake 分布式唯一 ID 算法 |
| 乐观锁 | 通过版本号或条件UPDATE实现的并发控制策略 |
| 业务状态码 | 6位字符串编码，格式 ABBCCC，在响应体中标识业务处理结果 |

## 二、业务范围与角色定义

### 2.1 名词解释（领域模型）

| 术语 | 说明 |
|------|------|
| 买家 (BUYER) | 普通消费者，可浏览商品、加入购物车、下单支付、发起退货投诉 |
| 卖家 (SELLER) | 店铺经营者，可开店、发布商品、管理订单、处理退货 |
| 管理员 (ADMIN) | 平台运营者，审核店铺/商品、处理投诉、仲裁争议、管理用户 |
| 店铺 (Store) | 卖家经营商品的载体，需经管理员审核通过后方可营业 |
| 商品 (Product) | 店铺中上架的商品，需经管理员审核通过后公开可见 |
| 订单 (Order) | 买家购买商品的记录，贯穿支付-发货-收货-完成生命周期 |
| 退货单 (Return) | 买家发起退货退款的工单，有仅退款和退货退款两种类型 |
| 投诉 (Complaint) | 买家的维权工单，可关联订单/商品/店铺/退货单 |
| 余额 (Balance) | 买家账户资金，用于下单支付；支持充值和退款入账 |
| 余额流水 (BalanceLog) | 每次余额变动的记录，含变动前后余额和关联单号 |

### 2.2 角色与权限（RBAC）

系统共有三种角色，角色编码如下：

| 编码 | 角色 | 说明 |
|------|------|------|
| 0 | BUYER（买家） | 默认注册角色 |
| 1 | SELLER（卖家） | 可申请开店 |
| 2 | ADMIN（管理员） | 平台管理 |

**权限矩阵**：

| 功能模块 | BUYER | SELLER | ADMIN |
|---------|-------|--------|-------|
| 浏览商品（已审核） | 可 | 可 | 可 |
| 搜索/查看商品详情 | 可 | 可 | 可 |
| 购物车管理 | 可 | — | — |
| 创建订单 / 支付 | 可 | — | — |
| 确认收货 | 可 | — | — |
| 发起退货 / 投诉 | 可 | — | — |
| 余额充值 / 查询流水 | 可 | 可 | 可 |
| 开设店铺 | — | 可（限1个） | — |
| 管理自己的店铺/商品 | — | 可 | — |
| 处理自己店铺的订单 | — | 可 | — |
| 审核自己店铺的退货 | — | 可 | — |
| 审核店铺 | — | — | 可 |
| 审核商品 | — | — | 可 |
| 管理所有用户 | — | — | 可 |
| 处理所有投诉 | — | — | 可 |
| 仲裁退货争议 | — | — | 可 |
| 管理分类 | — | — | 可 |

### 2.3 需求范围

**本次做**：
- 完整的买家-卖家-管理员三方电商协作流程
- 用户注册/登录（JWT 认证）
- 商品分类管理（两级树形结构）
- 店铺申请与审核
- 商品发布与审核（含多图）
- 购物车管理
- 下单支付（余额扣款，乐观锁防超扣）
- 订单生命周期管理（支付-发货-收货-完成-取消）
- 退货退款（仅退款 / 退货退款两种类型）
- 买家卖家消息沟通
- 投诉处理
- 退货争议自动关联投诉
- 账户余额充值与流水记录
- 基于角色的权限控制
- H2 内存数据库，启动即初始化

**本次不做**：
- 真实第三方支付对接（支付宝/微信）
- 短信/邮件通知
- 优惠券/满减促销
- 物流轨迹对接
- 数据埋点与报表
- 分布式部署与多活
- 生产级安全加固（限流、脱敏、防重放）

## 三、业务流程图

### 3.1 系统业务全景图

```
买家 (BUYER)                 卖家 (SELLER)                管理员 (ADMIN)
─────                       ─────                       ─────
注册/登录                    注册/登录                    登录
  │                           │                           │
浏览商品 ←────────────────── 发布商品 ────────────────→ 审核商品
  │                           │ (待审核)                   │
加入购物车                     │                           │
  │                           │ ←────────────────────── 审核通过
下单、余额扣款                 │                           │
  │                           │                           │
订单已支付 ──────────────────→ 发货 ───────────────────→ (可查看)
  │                           │                           │
确认收货                       │                           │
  │                           │                           │
订单完成                       │                           │
  │                           │                           │
发起退货 ────────────────────→ 审核退货                   │
  │                           │ (通过/拒绝)                │
  │                           │                           │
(仅退款→到账)                  │                           │
(退货退款→寄回→验货→退款)       │ ←── 争议 ──────────────→ 仲裁争议
  │                           │                           │
发起投诉 ──────────────────────────────────────────────→ 处理投诉
```

### 3.2 核心角色协作关系

买家、卖家、管理员围绕"店铺-商品-订单"三条主线展开协作：

1. **店铺线**：卖家申请开店 → 管理员审核 → 通过后对外营业
2. **商品线**：卖家发布商品 → 管理员审核 → 通过后买家可见可买
3. **订单线**：买家下单支付 → 卖家发货 → 买家收货 → 订单完成 → 退货/争议

## 四、非功能需求

### 4.1 性能需求

本项目定位为测试靶场，不设硬性 QPS/RT 指标。但作为参考：
- 单机 H2 内存数据库，所有查询均在毫秒级完成
- 无外部网络调用，所有业务操作均为本地事务
- 启动时间目标：<10 秒（含数据库初始化与种子数据写入）

### 4.2 可用性需求

- 单实例部署，无需高可用保障
- 重启即恢复初始状态（H2 内存数据库特性）
- 无需降级、熔断、异地多活

### 4.3 安全需求

| 安全措施 | 实现方式 |
|---------|---------|
| 密码存储 | BCrypt 单向哈希，数据库不存明文 |
| 身份认证 | JWT Bearer Token，24 小时过期，HS256 签名 |
| 权限控制 | 注解驱动 @RequireRole，拦截器校验 |
| 输入校验 | Jakarta Bean Validation (@NotBlank, @Size, @DecimalMin) |
| 文件上传限制 | 仅允许 image/* 类型，最大 10MB，UUID 文件名 |
| 数据隔离 | 卖家只能操作自己的店铺/商品/订单；买家只能操作自己的购物车/订单 |
| 并发控制 | 余额扣减和库存扣减使用乐观锁（条件 UPDATE） |

### 4.4 兼容性需求

- JDK 17+
- H2 数据库运行在 MySQL 兼容模式，DDL 可迁移至 MySQL
- 无老数据兼容需求（每次重启数据重置）

## 五、外部依赖与接口约定

### 5.1 依赖的第三方服务

**无**。本项目为零外部依赖设计：
- 数据库：H2 内存数据库（内嵌）
- 文件存储：本地文件系统 `./uploads/`
- 认证：自研 JWT 方案

### 5.2 对外提供的服务

为 API 测试用例生成智能体提供完整的 RESTful API。接口采用统一响应格式：

- 成功：`{ "code": "100000", "message": "success", "data": {...} }`
- 业务失败：`{ "code": "2BBCCC", "message": "错误描述", "data": null }`（HTTP 200）
- 系统异常：`{ "code": "300001", "message": "系统内部错误", "data": null }`（HTTP 500）

其中 `code` 为 6 位业务状态码，详见 API 参考文档。

## 六、上线与发布计划

### 6.1 数据初始化

系统首次启动时自动执行以下初始化：

1. **建表**：执行 `sql/schema.sql`，创建 12 张表（含中英文注释）
2. **种子数据**：`DataInitializer` 检测到数据库为空时自动写入：
   - 8 个用户（1 管理员 + 5 卖家 + 2 买家）
   - 22 个商品分类（5 顶级 + 17 二级）
   - 5 家店铺（已审核通过）
   - 60 个商品（每个店铺 12 个，已审核通过，含图片）

种子数据中的密码均经过 BCrypt 哈希处理，ID 由 MyBatis-Plus 雪花算法动态生成。

### 6.2 启动方式

```bash
cd foli-backend
./mvnw spring-boot:run
```

启动后访问：
- API 文档：http://localhost:8080/doc.html
- H2 控制台：http://localhost:8080/h2-console

### 6.3 回滚方案

由于使用 H2 内存数据库，重启即自动重置所有数据至初始状态。无需数据库备份或回滚脚本。
