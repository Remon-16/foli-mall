## 二、业务范围与角色定义

### 2.1 名词解释

| 术语 | 说明 |
|------|------|
| 会话 (Conversation) | 两个用户之间的消息通道，由会话 ID 唯一标识 |
| 会话 ID | 格式 `minUserId_maxUserId`，确定性排序，与发送顺序无关 |
| 投诉类型 | 包括 product_quality/service/delivery/fraud/return_dispute/other |
| 退货争议投诉 | return_dispute 类型，由卖家发起争议时系统自动创建 |

### 2.2 角色与权限

| 功能 | BUYER | SELLER | ADMIN |
|------|-------|--------|-------|
| 发送消息 | 可 | 可 | — |
| 查看自己的会话/消息 | 可 | 可 | — |
| 标记已读 | 可（自己的消息） | 可（自己的消息） | — |
| 发起投诉 | 可 | — | — |
| 查看自己的投诉 | 可 | — | — |
| 查看所有投诉 | — | — | 可 |
| 处理投诉 | — | — | 可 |

### 2.3 需求范围

**本次做**：买家卖家一对一消息；会话管理（分组+未读计数）；投诉创建与处理；退货争议自动创建投诉

**本次不做**：群聊/多人会话；消息推送/已读回执；投诉处理状态机的中间状态（当前 PENDING 可直接变为 RESOLVED/REJECTED，无需经过 IN_PROGRESS）

## 三、业务流程图

### 3.1 投诉状态机

```
  createComplaint
       │
       ▼
  ┌──────────┐
  │PENDING(0)│
  │  待处理   │
  └────┬─────┘
       │ handleComplaint
       │
       ├── RESOLVED(2) → 结束
       │
       └── REJECTED(3) → 结束

  IN_PROGRESS(1) — 已定义，但处理投诉时可直接从 PENDING 跳到 RESOLVED/REJECTED

  状态机校验：RESOLVED 或 REJECTED 后的投诉不可再处理
```

### 3.2 消息流程

```
买家 ── 发送消息 ──→ 卖家
  │                   │
  │  会话ID = min(buyerId, sellerId)_max(buyerId, sellerId)
  │                   │
  │  isRead = 0       │
  │                   │
  │  ←── 回复消息 ────│
  │                   │
  │  查看会话列表      │
  │  (按最新消息时间   │
  │   降序 + 未读数)   │
```

### 3.3 退货争议→投诉联动

```
卖家验货不通过 → 调用 dispute
  → 退货状态 → DISPUTED(7)
  → 自动创建 FmComplaint:
       userId = 店铺 owner
       storeId = 店铺 ID
       returnId = 退货 ID
       type = "return_dispute"
       status = PENDING(0)
  → 管理员前台看到投诉 → 处理投诉 → 仲裁退货
```

## 四、功能需求详细说明

### 4.1 发送消息

**前置条件**：已登录。

**业务规则**：
- 会话 ID 由两个用户 ID 确定性排序生成（min_max）
- 消息默认未读（isRead=0）
- 返回发送者和接收者昵称

### 4.2 会话列表

**业务规则**：
- 查询用户参与的所有消息（作为发送者或接收者）
- 按 conversationId 分组
- 每组返回：最新消息内容、未读计数（当前用户为接收者且未读）、对方用户昵称
- 按最新消息时间降序排列

### 4.3 标记已读

**业务规则**：
- 单条标记：验证消息的接收者是当前用户
- 批量标记：将整个会话中当前用户为接收者的所有未读消息设为已读

**异常分支**：消息不存在(210001) / 无权操作(200003)

### 4.4 创建投诉

**前置条件**：已登录。

**业务规则**：
- 可关联订单（orderId）、商品（productId）、店铺（storeId）、退货单（returnId）
- 投诉类型：product_quality / service / delivery / fraud / return_dispute / other
- 标题和内容非空
- 初始状态 PENDING(0)
- 不校验关联的订单/商品/退货是否存在

### 4.5 处理投诉

**前置条件**：ADMIN 角色。

**业务规则**：
- 状态机校验：已解决(RESOLVED)或已驳回(REJECTED)的投诉不可再处理
- 记录处理人、处理结果、处理时间
- 目标状态可以是 RESOLVED(2) 或 REJECTED(3)

**异常分支**：投诉不存在(209001) / 已处理完毕(209002)

## 五、非功能需求

- 消息内容无长度限制（当前实现）
- 会话列表查询为内存聚合（非数据库 GROUP BY），数据量大时需优化
- 退货争议自动创建的投诉在管理员仲裁后不会自动更新状态
